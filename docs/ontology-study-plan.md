# 온톨로지 공부 구조와 실행 플랜

## 한 줄 목표

`어머니와의 실제 대화`에서 나오는 건강/생활 신호를 `일관된 의미 체계`로 정의하고,
그 의미 체계를 나중에 `저장`, `요약`, `알림`, `검색`, `분석`에 재사용할 수 있게 만드는 것이 목표다.

## 먼저 기준을 바로 세우기

지금 단계에서 말하는 온톨로지는 학술적인 RDF/OWL 중심 온톨로지보다,
이 서비스 안에서 같은 의미를 계속 같은 방식으로 쓰게 만드는 `도메인 의미 체계`에 가깝다.

즉, 지금 먼저 공부해야 하는 것은 아래다.

- 어떤 개념이 독립 엔티티인지
- 어떤 것은 시간에 따라 바뀌는 관찰값인지
- 어떤 것은 추론 결과인지
- 어떤 것은 가족에게 전달할 행동 기준인지
- 같은 말을 어떤 기준으로 같은 의미로 묶을지
- 불확실하거나 잘못 들은 경우를 어떻게 남길지

반대로 지금 당장 깊게 파지 않아도 되는 것은 아래다.

- RDF, OWL, Protégé
- 그래프 데이터베이스 설계
- 의료 표준 전체 암기
- 복잡한 추론 엔진

이건 실제 대화 데이터가 쌓이고 의미 체계가 안정된 뒤에 해도 늦지 않다.

## 공부 구조

온톨로지 공부는 아래 5개 층으로 나누는 게 가장 좋다.

### 1. 원본 층

여기서는 `실제로 무슨 말이 오갔는가`만 본다.

- 세션
- 연결
- 턴
- 사용자 전사
- 가준이 응답
- 추출 시점의 원문 근거

핵심 질문:

- 어떤 단위를 나중에 다시 읽고 검증할 수 있어야 하나
- 잘못 들은 전사도 원본 사실로 남길 것인가
- 부분 전사와 최종 전사를 어떻게 구분할 것인가

### 2. 관찰 층

여기서는 `대화에서 직접 확인된 사실`만 본다.

- 통증 있음/없음
- 통증 부위
- 식사 상태
- 수면 상태
- 보청기 착용 여부
- 활동 가능 여부
- 농사 가능 여부
- 어지럼/낙상 여부

핵심 질문:

- `언급 없음`, `부정`, `긍정`을 어떻게 구분할 것인가
- 이 신호는 사용자 말에 직접 나온 것인가, 모델이 추정한 것인가
- 한 턴에 여러 신호가 동시에 나올 수 있을 때 기준은 무엇인가

### 3. 해석 층

여기서는 `관찰값을 서비스 의미로 어떻게 해석할 것인가`를 본다.

- watch / urgent
- 가족 확인 필요
- 하루 상태 변화
- 반복 통증
- 식사 저하 지속

핵심 질문:

- 어떤 규칙이 단순 관찰을 위험 신호로 바꾸는가
- 위험은 즉시성인가, 누적성인가
- 한 번의 발화보다 하루 단위가 더 중요한 신호는 무엇인가

### 4. 행동 층

여기서는 `이 해석을 가지고 무엇을 할 것인가`를 정한다.

- 하루 요약 생성
- 가족 알림 생성
- 추후 상담/전화 권고
- 대시보드 표시
- 검색/리뷰 대상 지정

핵심 질문:

- 누구에게 어떤 형식으로 전달할 것인가
- watch 와 urgent 의 차이는 실제 행동에서 무엇인가
- 하루 요약과 즉시 알림은 어떤 정보를 다르게 담아야 하나

### 5. 버전/검증 층

여기서는 `이 의미 체계가 시간이 지나도 유지되게` 만드는 장치다.

- extractor version
- ontology version
- evidence
- correction log
- annotation guide

핵심 질문:

- 나중에 기준이 바뀌면 과거 데이터와 어떻게 비교할 것인가
- 사람이 고친 값과 자동 추출 값을 어떻게 구분할 것인가
- 정확도를 어떤 데이터셋으로 측정할 것인가

## 당신이 공부해야 하는 순서

공부 순서는 `데이터 모델링 -> 의미 정의 -> 추출 평가 -> 요약/알림 설계 -> 표준 매핑`이 맞다.

이 순서를 뒤집으면 너무 쉽게 추상화 과잉에 빠진다.

### 1단계. 엔티티 / 이벤트 / 상태 구분

먼저 아래 3개를 분리해서 이해해야 한다.

- 엔티티: 사람, 가족, 기기, 세션
- 이벤트: 한 턴, 낙상 언급, 보청기 권유
- 상태: 식사 상태, 수면 상태, 통증 상태

이걸 명확히 구분해야 나중에 테이블이 흔들리지 않는다.

### 2단계. 관찰과 추론 구분

매우 중요하다.

- 관찰: “무릎이 아파”
- 추론: `pain_present = true`
- 서비스 판단: `risk_level = watch`

이 3개를 섞으면 나중에 왜 그런 판단이 나왔는지 설명이 안 된다.

### 3단계. 불확실성 모델링

지금 서비스에서는 STT 오류가 실제 핵심 리스크다.
그래서 온톨로지 공부에서 `불확실성`은 필수다.

반드시 구분해야 한다.

- 언급 없음
- 잘 안 들림
- 모델이 다시 물음
- 사용자 발화가 모호함
- 규칙 기반 추출
- LLM 기반 추출
- 사람이 확인해 수정함

### 4단계. 시간 축 모델링

건강 신호는 대부분 시간에 따라 바뀐다.
그래서 `한 턴`, `하루`, `장기 프로필`을 나눠서 봐야 한다.

- 턴: 그 순간 나온 신호
- 하루: 그날의 요약 상태
- 장기: 최근 7일/30일 경향

### 5단계. 통제어휘와 동의어 묶기

예를 들어 아래는 사실 같은 의미로 묶일 수 있다.

- 아파요 / 쑤셔요 / 결려요 / 시큰해요
- 밥 먹었어 / 조금 먹었어 / 못 먹었어
- 어지러워 / 핑 돌았어

온톨로지는 결국 `말의 다양성`을 `의미의 일관성`으로 바꾸는 작업이다.

## 4주 실행 플랜

### 1주차: 실제 데이터 중심으로 개념 수집

목표:
`무슨 개념이 실제로 자주 나오는지`부터 잡는다.

공부 주제:

- 도메인 모델링 기본
- 엔티티 vs 이벤트 vs 상태
- 관찰 vs 추론 구분

실행:

1. 실제 대화 50~100턴을 모은다.
2. 각 턴에서 반복 등장하는 개념을 표시한다.
3. 비슷한 의미를 묶는다.
4. 잘 안 들린 사례를 따로 분류한다.

산출물:

- concept inventory
- ambiguous transcript list
- first-pass signal list

완료 기준:

- “이 서비스에서 반복적으로 다뤄야 할 건강/생활 개념” 목록이 나온다.

### 2주차: 신호 사전 만들기

목표:
각 건강 신호를 `정의`, `값 범위`, `근거`, `예외`까지 포함해 정리한다.

공부 주제:

- controlled vocabulary
- enum 설계
- null / false / true 의미 설계
- evidence / provenance 개념

실행:

1. 신호별 정의 문서를 만든다.
2. 각 신호의 허용값을 정한다.
3. 예문과 반례를 붙인다.
4. alert 에 직접 쓰일지, summary 에만 쓰일지 표시한다.

산출물:

- signal dictionary v1
- evidence rule v1
- value normalization table

완료 기준:

- 각 필드에 대해 “언제 채우고 언제 비우는지”가 문장으로 설명 가능해야 한다.

### 3주차: 추출과 평가 구조 설계

목표:
자동 추출이 얼마나 맞는지 평가할 수 있는 구조를 만든다.

공부 주제:

- annotation guide
- precision / recall
- confusion case 분석
- versioning

실행:

1. 50턴 정도를 손으로 라벨링한다.
2. 규칙 기반 추출 결과와 비교한다.
3. 어떤 신호가 자주 틀리는지 본다.
4. `review_status`, `extraction_version`, `evidence`의 필요성을 정리한다.

산출물:

- annotation guide v1
- gold sample set
- extraction error notes

완료 기준:

- “무엇이 틀렸는지”를 감으로 말하지 않고 유형으로 말할 수 있어야 한다.

### 4주차: 요약/알림 의미 체계 확정

목표:
관찰 신호를 가족 전달 형식으로 바꾸는 규칙을 만든다.

공부 주제:

- summary design
- threshold design
- escalation rule
- explainability

실행:

1. 하루 요약 포맷을 고정한다.
2. watch / urgent 기준을 표로 만든다.
3. 어떤 신호 조합이 가족 전달로 이어지는지 정한다.
4. 요약문에 어떤 근거를 남길지 정한다.

산출물:

- daily summary spec v1
- alert rule matrix v1
- ontology version 0.1

완료 기준:

- 하루 요약과 즉시 알림이 서로 다른 역할을 갖도록 정의되어 있어야 한다.

## 추천 산출물 구조

문서를 아래 구조로 관리하면 좋다.

```text
docs/ontology/
  01-concept-inventory.md
  02-entity-event-state-model.md
  03-signal-dictionary.md
  04-annotation-guide.md
  05-alert-rule-matrix.md
  06-daily-summary-spec.md
  07-ontology-version-log.md
```

각 문서는 아래 질문에 답해야 한다.

- 정의가 무엇인가
- 값 범위는 무엇인가
- 근거는 무엇인가
- 반례는 무엇인가
- 누가 사용하는가
- 어떤 기능에서 쓰이는가
- 바뀌면 어떤 데이터가 영향 받는가

## 지금 당신에게 가장 중요한 공부 포인트

모든 걸 다 공부할 필요는 없다.
지금 가장 중요한 것은 아래 6개다.

1. `관찰`과 `판단`을 분리하는 법
2. `unknown`과 `false`를 분리하는 법
3. `한 턴 정보`와 `하루 정보`를 분리하는 법
4. `자연어 다양성`을 `정규값`으로 매핑하는 법
5. `불확실성`과 `근거`를 남기는 법
6. `나중에 기준이 바뀌어도 버틸 수 있게 versioning 하는 법`

## 나중에 공부하면 좋은 것

아래는 v1이 안정된 뒤에 들어가면 된다.

- HL7 FHIR 의 Observation 식 사고방식
- SNOMED CT 같은 표준 용어 체계의 개념
- RDF/OWL 기반 온톨로지 모델링
- knowledge graph / graph DB
- 임상 코딩 표준과 매핑 전략

중요한 건, 이걸 먼저 공부하는 게 아니라
`실제 어머니 대화 데이터에서 의미 체계를 안정화한 다음`
표준과 매핑하는 순서가 맞다는 점이다.

## 실전 운영 원칙

- 온톨로지는 책상에서 완성되지 않는다.
- 실제 대화 100턴을 본 사람의 정의가, 추상 이론보다 훨씬 낫다.
- 필드는 적게 시작하고 정의를 엄격하게 하는 편이 좋다.
- 기준이 바뀌는 건 정상이다. 대신 버전과 근거를 남겨야 한다.
- 지금은 “완벽한 의료 모델”보다 “가족에게 전달 가능한 일관된 의미 체계”가 더 중요하다.

## 최종 목표 상태

온톨로지 공부가 잘 끝났다는 것은 아래가 가능한 상태다.

- 같은 의미를 항상 같은 구조로 저장할 수 있다.
- 왜 `watch` 또는 `urgent` 인지 설명할 수 있다.
- 하루 요약이 사람마다 흔들리지 않는다.
- 규칙 기반 추출과 LLM 추출을 같은 기준으로 비교할 수 있다.
- 나중에 표준 체계와 매핑할 때도 현재 데이터를 버리지 않아도 된다.
